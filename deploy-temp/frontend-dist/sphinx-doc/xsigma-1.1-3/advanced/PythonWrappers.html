Python Wrappers#

Introduction#

This document is a reference for using XSIGMA from Python. It is not a tutorial and provides very little information about XSIGMA itself, but instead describes in detail the features of the Python wrappers and how using XSIGMA from Python differs from using XSIGMA from C++. It assumes that the reader is already somewhat familiar with both Python and XSIGMA.

Background#

The Python wrappers are automatically generated from the XSIGMA source code, and for the most part, there is a one-to-one mapping between the XSIGMA classes and methods that you can use from Python and the ones that you can use from C++. More specifically, the wrappers are a package of Python extension modules that interface directly to the XSIGMA C++ libraries. When you use XSIGMA through the wrappers, you are actually executing compiled C++ code, and there is very little performance difference between XSIGMA/C++ and XSIGMA/Python.

Installation#

XSIGMA for Python can be installed via either conda or pip, where the conda packages is maintained on conda-forge, while the pip packages are maintained by the XSIGMA developers themselves. If you are first getting started, then pip is probably the most convenient way to install XSIGMA for Python:

pip install xsigma

This will provide a basic installation of XSIGMA that includes all core functionality, but which will not include some of the specialized XSIGMA modules that rely on external libraries. Binary packages for XSIGMA can also be downloaded directly from https://www.xsigma.org/download/.

Instructions for building XSIGMA from source code are given in the file Documentation/dev/build.md within the source repository.

Importing#

XSIGMA is comprised of over one hundred individual modules. Programs can import just the modules that are needed, in order to reduce load time.

from xsigmamodules.xsigmaCommonCore import xsigmaObject
from xsigmamodules.xsigmaFiltersSources import xsigmaConeSource, xsigmaSphereSource
from xsigmamodules.xsigmaRenderingCore import (
    xsigmaActor,
    xsigmaDataSetMapper,
    xsigmaRenderer,
    xsigmaRenderWindow
)
import xsigmamodules.xsigmaRenderingOpenGL2

When getting started, however, it is hard to know what modules you will need. So if you are experimenting with XSIGMA in a Python console, or writing a quick and dirty Python script, it is easiest to simply import everything. There is a special module called ‘all’ that allows this to be done:

from xsigmamodules.all import *

After importing the XSIGMA classes, you can check to see which module each of the classes comes from:

for c in xsigmaObject, xsigmaConeSource, xsigmaRenderWindow:
    print(f"from {c.__module__} import {c.__name__}")

The output is as follows:

from xsigmamodules.xsigmaCommonCore import xsigmaObject
from xsigmamodules.xsigmaFiltersSources import xsigmaConeSource
from xsigmamodules.xsigmaRenderingCore import xsigmaRenderWindow

Factories and Implementation Modules#

In the first ‘import’ example above, you might be wondering about this line:

import xsigmamodules.xsigmaRenderingOpenGL2

This import is needed because xsigmaRenderingOpenGL2 provides the OpenGL implementations of the classes in xsigmaRenderingCore. To see this in action, open a new Python console and do the following:

>>> from xsigmamodules.xsigmaRenderingCore import xsigmaRenderWindow
>>> renwin = xsigmaRenderWindow()
>>> type(renwin)
<class 'xsigmamodules.xsigmaRenderingCore.xsigmaRenderWindow'>
>>>
>>> import xsigmamodules.xsigmaRenderingOpenGL2
>>> renwin2 = xsigmaRenderWindow()
>>> type(renwin2)
<class 'xsigmamodules.xsigmaRenderingOpenGL2.xsigmaXOpenGLRenderWindow'>

After xsigmaRenderingOpenGL2 has been imported, the xsigmaRenderWindow() constructor magically starts returning a different type of object. This occurs because xsigmaRenderWindow is a factory class, which means that the kind of object it produces can be overridden by an implementation class. In order for the implementation class to do the override, all that is necessary is that its module is imported. To make things even more confusing, xsigmaRenderingOpenGL2 is not the only module that contains implementations for the factory classes in xsigmaRenderingCore. The following modules are often needed, as well:

import xsigmamodules.xsigmaInteractionStyle
import xsigmamodules.xsigmaRenderingFreeType

Although you only need implementations for the factory classes that you use, it can be hard to know which classes are factory classes, or what modules contain implementations for them. Also, it can be difficult to even know what classes you are using, since many XSIGMA classes make use of other XSIGMA classes. An example of this is xsigmaDataSetMapper, which internally uses xsigmaPolyDataMapper to do the rendering. So even though xsigmaDataSetMapper is not a factory class, it needs an OpenGL implementation for xsigmaPolyDataMapper.

The simplest approach is to import all the important implementation modules into your program, even if you are not certain that you need them.

  • For xsigmaRenderingCore, import xsigmaRenderingOpenGL2, xsigmaRenderingFreeType, xsigmaInteractionStyle

  • For xsigmaRenderingVolume, import xsigmaRenderingVolumeOpenGL2

  • For xsigmaCharts, import xsigmaContextOpenGL2

Classic XSIGMA Import#

There are many XSIGMA programs that still import the ‘xsigma’ module, which has been available since XSIGMA 4.0, rather than using the ‘xsigmamodules’ package that was introduced in XSIGMA 8.2:

import xsigma

The advantage (and disadvantage) of this is that it imports everything. It requires just one import statement for all of XSIGMA, but it can be slow because XSIGMA has grown to be very large over the years.

Also note that, between XSIGMA 8.2 and XSIGMA 9.2.5, the use of the xsigma module would confuse the auto-completion features of IDEs such as PyCharm. This was fixed in XSIGMA 9.2.6. For 9.2.5 and earlier, the following can be used:

import xsigmamodules.all as xsigma

From the programmer’s perspective, this is equivalent to ‘import xsigma’.

XSIGMA Classes and Objects#

Classes Derived from xsigmaObjectBase#

In C++, classes derived from xsigmaObjectBase are instantiated by calling New(). In Python, these classes are instantiated by simply calling the constructor:

o = xsigmaObject()

For factory classes, the returned object’s type might be a subtype of the class. This occurs because the Python wrappers are actually calling New() for you, which allows the XSIGMA factory overrides to occur:

>>> a = xsigmaActor()
>>> type(a)
<class 'xsigmamodules.xsigmaRenderingOpenGL2.xsigmaOpenGLActor'>

When you create a XSIGMA object in Python, you are in fact creating two objects: a C++ object, and a Python object that holds a pointer to the C++ object. The repr() of the object shows the memory address of the C++ object (in parentheses) and of the Python object (after the ‘at’):

>>> a = xsigmaFloatArray()
>>> a
<xsigmamodules.xsigmaCommonCore.xsigmaFloatArray(0x5653a6a6f700) at 0x7f0e7aecf5e0>

If you call str() or print() on these objects, the wrappers will call the C++ PrintSelf() method. The printed information can be useful for debugging:

>>> o = xsigmaObject()
>>> print(o)
xsigmaObject (0x55858308a210)
  Debug: Off
  Modified Time: 85
  Reference Count: 1
  Registered Events: (none)

Other Classes (Special Types)#

XSIGMA also uses several classes that aren’t derived from xsigmaObjectBase. The most important of these is xsigmaVariant, which can hold any type of object:

>>> v1 = xsigmaVariant('hello')
>>> v1
xsigmamodules.xsigmaCommonCore.xsigmaVariant('hello')
>>> v2 = xsigmaVariant(3.14)
>>> v2
xsigmamodules.xsigmaCommonCore.xsigmaVariant(3.14)

The wrapping of these classes is fully automatic, but is done in a slightly different manner than xsigmaObjectBase-derived classes. First, these classes have no New() method, and instead the public C++ constructors are wrapped to create an equivalent Python constructor. Second, the Python object contains its own copy of the C++ object, rather than containing just a pointer to the C++ object. The vast majority of these classes are lightweight containers and numerical types. For example, xsigmaQuaterniond, xsigmaRectf, xsigmaColor4ub, etc. Many of them are actually class templates, which are discussed below.

When you apply print() or str() to these objects, the operator<< of the underlying C++ object is used to print them. For repr(), the name of the type name is printed, followed by the str() output in prentheses. The result looks similar to a constructor, though it might look strange depending on what operator<< produces.

>> v = xsigmaVariant()
>> print(repr(v))
xsigmamodules.xsigmaCommonCore.xsigmaVariant((invalid))

Class Templates#

There are several C++ templates in XSIGMA, which can be tricky to use from the wrappers since the Python language has no real concept of templates. The wrappers wrap templates as dictionary-like objects that map the template parameters to template instantiations:

>>> xsigmaSOADataArrayTemplate
<template xsigmaCommonCorePython.xsigmaSOADataArrayTemplate>
>>> xsigmaSOADataArrayTemplate.keys()
['char', 'int8', 'uint8', 'int16', 'uint16', 'int32', 'uint32', 'int',
'uint', 'int64', 'uint64', 'float32', 'float64']
>>> c = xsigmaSOADataArrayTemplate['float64']
>>> c
<class 'xsigmamodules.xsigmaCommonCore.xsigmaSOADataArrayTemplate_IdE'>

The wrappers instantiate the C++ template for a few useful types, as indicated by the keys() of the template. The Python type name also has a suffix (the ‘IdE’) that indicates the template parameters in a compressed form according to IA64 C++ ABI name mangling rules, even when XSIGMA is built with a compiler that does not use the IA64 ABI natively.

Objects are created by first instantiating the template, and then instantiating the class:

>>> a = xsigmaSOADataArrayTemplate['float32']()
>>> a.SetNumberOfComponents(3)

In the case of multiple template parameters, the syntax can look rather complicated, but really it isn’t all that bad. For example, constructing a xsigmaTuple<double,4> in Python looks like this, with the template args in square brackets and the constructor args in parentheses:

>>> xsigmaTuple['float64',4]([1.0, 2.0, 3.0, 4.0])
xsigmamodules.xsigmaCommonMath.xsigmaTuple_IdLi4EE([1.0, 2.0, 3.0, 4.0])

The type names are the same as numpy’s dtypes: bool, int8, uint8, int16, uint16, int32, uint32, int64, uint64, float32, and float64. Since int64 is ‘long long’, int is used for long. Also see Template Keys in Advanced Topics.

Method Calls#

When XSIGMA methods are called from Python, conversion of all parameters from Python to C++ occurs automatically. That is, if the C++ method signature expects an integral type, you can pass a Python int, and if C++ expects a floating-point type, you can pass a Python float (or any type that allows implicit conversion to float).

For C++ ‘char’ parameters, which are rarely used in XSIGMA, you must pass a string with a length of 1. For unicode, the code must fit into eight bits (either ASCII, or within the Latin-1 Supplement block). A null character is represented by ‘\0’, or equivalently chr(0).

A Python tuple, list, or any other Python sequence can be passed to a XSIGMA method that requires an array or std::vector in C++:

>>> a = xsigmaActor()
>>> p = (100.0, 200.0, 100.0)
>>> a.SetPosition(p)

If the method is going to modify the array that you pass as a parameter, then you must pass a Python list that has the correct number of slots to accept the returned values. If you try this with a tuple, you will get a TypeError because tuple is immutable.

>>> z = [0.0, 0.0, 0.0]
>>> xsigmaMath.Cross((1,0,0),(0,1,0),z)
>>> print(z)
[0.0, 0.0, 1.0]

For multi-dimensional array parameters, you can either use a nested list, or you can use numpy array with the correct shape.

If the C++ method returns a pointer to an array, then in Python the method will return a tuple if the wrappers know the size of the array. In most cases, the size is hinted in the header file.

>>> a = xsigmaActor()
>>> print(a.GetPosition())
(0.0, 0.0, 0.0)

Finally, Python None is treated the same as C++ nullptr, which allows you to pass null objects and null strings:

>>> a = xsigmaActor()
>>> a.SetMapper(None)
>>> print(a.GetMapper())
None

Wrappable and Unwrappable Methods#

A method cannot be used from Python if its C++ parameters or return type cannot be converted to or from Python by the wrappers, or if the method is templated. Common non-convertible types include std::ostream, std::istream, and all STL container types except for std::vector (see below), and any non-trivial pointer type or any pointer to an object whose class is not derived from xsigmaObjectBase.

The wrappable parameter types are:

  • char, wrapped as a single ASCII character in a Python str

  • signed char and unsigned char, wrapped as Python int

  • short, int, long and long long, wrapped as Python int

  • unsigned short to unsigned long long, wrapped as Python int

  • float and double, wrapped as Python float

  • size_t and ssize_t, wrapped as Python int

  • std::string, wrapped as Python str via utf-8 encoding/decoding

  • typedefs of all the above, for any typedef defined in a XSIGMA header file

  • std::vector<T> where T is one of the above, as Python tuple or list

  • const T& where T is any of the above, wrapped as described above

  • T[N] where T is a fundamental type, as Python tuple or list

  • T[N][M] where T is a fundamental type, as nested tuple or list

  • T* where T is a fundamental type, as tuple or list

  • xsigmaObjectBase* and derived types, as their respective Python type

  • xsigmaSmartPointer<T> as the Python xsigmaObjectBase-derived type T

  • std::vector<xsigmaSmartPointer<T>> as a sequence of objects of type T

  • const std::vector<xsigmaSmartPointer<T>> as a sequence of objects of type T

  • other wrapped classes (like xsigmaVariant), but not pointers to these types

  • char*, as Python str via utf-8 encoding/decoding

  • void*, as Python buffer (e.g. bytes or bytearray)

  • the parameter list (void (*f)(void*), void*) as a Python callable type

References like int& and std::string& are wrapped via a reference proxy type as described in the Pass by Reference section below. Non-const references to std::vector<T> and other mutable types do not use a proxy, but instead require that a mutable Python object is passed, for example a list rather than a tuple.

A void* parameter can accept a pointer in two different ways: either from any Python object that supports the Python buffer protocol (this includes all numpy arrays along with the Python bytes and bytearray types), or from a string that contains a mangled pointer of the form ‘_hhhhhhhhhhhh_p_void’ where ‘hhhhhhhhhhhh’ is the hexadecimal address. Return-value void* will always be a string containing the mangled pointer.

Also, a T* parameter for fundamental type T can accept a buffer object, if and only if it is annotated with the XSIGMA_ZEROCOPY hint in the header file. With this hint, a numpy array of T can be passed to a T* parameter and the XSIGMA method will directly access the memory buffer of the array. Hence the name ‘zerocopy’, which indicates no copying is done, and that direct memory access is used.

The xsigmaObject::AddObserver() method has a special wrapping, as discussed in the Observer Callbacks section below.

Conversion Constructors#

If a wrapped type has constructor that takes one parameter, and if that constructor is not declared ‘explicit’, then the wrappers will automatically use that constructor for type conversion to the parameter type. The wrappers ensure that this conversion occurs in Python in the same manner that it is expected to occur in C++.

For example, xsigmaVariantArray has a method InsertNextItem(v:xsigmaVariant), and xsigmaVariant has a constructor xsigmaVariant(x:int). So, you can do this:

>>> variantArray.InsertNextItem(1)

The wrappers will automatically construct a xsigmaVariant from ‘1’, and will then pass it as a parameter to InsertNextItem(). This is a feature that most C++ programmers will take for granted, but Python users might find it surprising.

Overloaded Methods#

If you call a XSIGMA method that is overloaded, the Python wrappers will choose the overload that best matches the supplied arguments. This matching takes into account all allowed implicit conversions, such as int to float or any conversion constructors that are defined for wrapped objects.

Some overloads will be unavailable (not wrapped) either because they are unwrappable as per the criteria described above, or because they are shadowed by another overload that is always preferable. A simple example of this is any methods that is overloaded on C++ float and double. The Python float type is a perfect match C++ double, therefore the float overload is not wrapped.

Static Methods#

A static method can be called without an instance. For example,

xsigmaObject.SetGlobalWarningDisplay(1)

Some XSIGMA classes, like xsigmaMath, consist solely of static methods. For others, like xsigmaMatrix4x4, most of the non-static methods have static overloads. Within Python, the only way to tell if a XSIGMA method is static (other than trying it) is to look at its docstring.

Unbound Methods#

When a non-static method is called on the class, rather than on an instance, it is called an unbound method call. An unbound method call must provide ‘self’ as the first argument, where ‘self’ is an instance of either the class or a subclass.

w = xsigmaRenderWindow()
xsigmaWindow.Render(w)

In other words, the wrappers translate Python unbound method calls into C++ unbound method calls. These are useful when deriving a Python class from a wrapped XSIGMA class, since they allow you to call any base class methods that have been overridden in the subclass.

Operator Methods#

For special classes (the ones not derived from xsigmaObjectBase), some useful C++ operators are wrapped in python. The ‘[]’ operator is wrapped for indexing and item assignment, but because it relies on hints to guess which indices are out-of-bounds, it is only wrapped for xsigmaVector and related classes.

The comparison operators ‘<’ ‘<=’ ‘==’ ‘>=’ ‘>’ are wrapped for all classes that have these operators in C++. These operators allow sorting of xsigmaVariant objects with Python.

The ‘<<’ operator for printing is wrapped and is used by the python print() and str() commands.

Strings and Bytes#

XSIGMA uses both char* and std::string for strings. As far as the wrappers are concerned, these are equivalent except that the former can be nullptr (None in Python). For both, the expected encoding is ASCII or utf-8.

In Python, either str or bytes can be used to store strings, and both of these can be passed to XSIGMA methods that require char* or std::string (or the legacy xsigmaStdString). A str object is passed to XSIGMA as utf-8, while a bytes object is passed as-is.

When a XSIGMA method returns a string, it is received in Python as a str object if it is valid utf-8, or as a bytes object if not. The caller should check the type of the returned object (str, bytes, or perhaps None) if there is any reason to suspect that non-utf-8 text might be present.

STL Containers#

XSIGMA provides conversion between std::vector and Python sequences such as tuple and list. If the C++ method returns a vector, the Python method will return a tuple:

C++: const std::vector<std::string>& GetPaths()
C++: std::vector<std::string> GetPaths()
Python: GetPaths() -> Tuple[str]

If the C++ method accepts a vector, then the Python method can be passed any sequence with compatible values:

C++: void SetPaths(const std::vector<std::string>& paths)
C++: void SetPaths(std::vector<std::string> paths)
Python: SetPaths(paths: Sequence[str]) -> None

Furthermore, if the C++ method accepts a non-const vector reference, then the Python method can be passed a mutable sequence (e.g. list):

C++: void GetPaths(std::vector<std::string>& paths)
Python: GetPaths(paths: MutableSequence[str]) -> None

The value type of the std::vector<T> must be std::string or a fundamental numeric type such as double or int (including signed char and unsigned char but excluding char).

Smart pointers#

The wrappers will automatically convert between C++ xsigmaSmartPointer<T> and objects of type T (or None, if the smart pointer is empty):

C++: xsigmaSmartPointer<xsigmaObject> TakeObject()
Python: TakeObject() -> xsigmaObject

In other words, in Python the smart pointer doesn’t look any different from the object it points to. Under the hood, however, the wrappers understand that the smart pointer carries a reference to the object and will take responsibility for deleting that reference.

A C++ method can return a vector of smart pointers, which will be seen in Python as a tuple of objects:

C++: std::vector<xsigmaSmartPointer<xsigmaObject>> GetObjects()
Python: GetObject() -> Tuple[xsigmaObject]

If a C++ method expects std::vector<xsigmaSmartPointer<T>> as a parameter, the wrappers will automatically construct the vector from any sequence that is passed from Python. The objects in the sequence must be of type T (or a subclass of T, or None). If not, a TypeError will be raised.

Pass by Reference#

Many XSIGMA methods use pass-by-reference to return values back to the caller. Calling these methods from Python requires special consideration, since Python’s str, tuple, int, and float types are immutable. The wrappers provide a ‘reference’ type, which is a simple container that allows pass-by-reference.

For example, consider the following C++ method that uses pass-by-reference:

void GetCellAtId(xsigmaIdType cellId, xsigmaIdType& cellSize, xsigmaIdType const*& cellPoints)

It requires a reference to xsigmaIdType (a Python int), and to xsigmaIdType const* (a tuple of ints). So we can call this method as follows:

>>> from xsigmamodules.xsigmaCommonCore import reference
>>> from xsigmamodules.xsigmaCommonDataModel import xsigmaCellArray
>>>
>>> # Build a cell array
>>> a = xsigmaCellArray()
>>> a.InsertNextCell(3, (1, 3, 0))
>>>
>>> # Create the reference objects
>>> n = reference(0)
>>> t = reference((0,))
>>>
>>> # Call the pass-by-reference method
>>> a.GetCellAtId(0, n, t)
>>>
>>> n.get()
3
>>> t.get()
(1, 3, 0)

Some important notes when using pass-by-reference:

  1. The reference constructor must be given a value of the desired type. The method might use this value or might ignore it.

  2. Calling the get() method of the reference is usually unnecessary, because the reference already supports the interface protocols of the object that it contains.

Preconditions#

One very real concern when using XSIGMA from Python is that the parameters that you pass to a method might cause the program to crash. In particular, it is very easy to pass an index that causes an out-of-bounds memory access, since the C++ methods don’t do bounds checking. As a safety precaution, the wrappers perform the bounds check before the C++ method is called:

>>> a = xsigmaFloatArray()
>>> a.GetValue(10)
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
ValueError: expects 0 <= id && id < GetNumberOfValues()

All precondition checks raise a ValueError if they fail, since they are checks on the values of the parameters. The wrappers don’t know if C++ is using the parameter as an index, so IndexError is not used.

Currently the only way to find out if a method has preconditions is to look at the declaration of the method in the C++ header file to see if it has a XSIGMA_EXPECTS hint.

Observer Callbacks#

Similar to what can be done in C++, a Python function can be called each time a XSIGMA event is invoked on a given object. In general, the callback function should have the signature func(obj:xsigmaObject, event:str), or func(self, obj:xsigmaObject, event:str) if it is a method of a class.

>>> def onObjectModified(object, event):
>>>     print('object: %s - event: %s' % (object.GetClassName(), event))
>>>
>>> o = xsigmaObject()
>>> o.AddObserver(xsigmaCommand.ModifiedEvent, onObjectModified)
1
>>> o.Modified()
object: xsigmaObject - event: ModifiedEvent

Call Data#

In case there is a ‘CallData’ value associated with an event, in C++, you have to cast it from void* to the expected type using reinterpret_cast. The equivalent in python is to add a CallDataType attribute to the associated python callback method. The supported CallDataType values are XSIGMA_STRING, XSIGMA_OBJECT, XSIGMA_INT, XSIGMA_LONG, XSIGMA_DOUBLE, and XSIGMA_FLOAT.

The following example uses a function as a callback, but a method or any callable object can be used:

>>> from xsigmamodules.xsigmaCommonCore import xsigmaCommand, XSIGMA_INT
>>>
>>> def onError(object, event, calldata):
>>>     print('object: %s - event: %s - msg: %s' % (object.GetClassName(), event, calldata))
>>>
>>> onError.CallDataType = XSIGMA_INT
>>>
>>> lt = xsigmaLookupTable()
>>> lt.AddObserver(xsigmaCommand.ErrorEvent, onError)
1
>>> lt.SetTableRange(2,1)
object: xsigmaLookupTable - event: ErrorEvent - msg: ERROR:
In /home/user/XSIGMA/Common/Core/xsigmaLookupTable.cxx, line 122
xsigmaLookupTable (0x6b40b30): Bad table range: [2, 1]

For convenience, the CallDataType can also be specified where the function is first declared with the help of the @calldata_type decorator:

>>> from xsigmamodules.util.misc import calldata_type
>>>
>>> @calldata_type(XSIGMA_INT)
>>> def onError(object, event, calldata):
>>>     print('object: %s - event: %s - msg: %s' % (object.GetClassName(),
                                                    event, calldata))

Other Wrapped Entities#

Constants#

Most of the constants defined in the XSIGMA header files are available in Python, and they can be accessed from the module in which they are defined. Many of these are found in the xsigmaCommonCore module, where they were defined as preprocessor macros.

>>> from xsigmamodules.xsigmaCommonCore import XSIGMA_DOUBLE_MAX
>>> XSIGMA_DOUBLE_MAX
1.0000000000000001e+299

Others are defined as enums, often within a class namespace. If the enum is anonymous, then its values are int.

>>> xsigmaCommand.ErrorEvent
39

Constants in the header files are wrapped if they are enums, or if they are const variables of a wrappable scalar type, or if they are preprocessor symbols that evaluate to integer, floating-point, or string literal types.

Enum Types#

Each named enum type is wrapped as a new Python type, and members of the enum are instances of that type. This allows type checking for enum types:

>>> from xsigmamodules.xsigmaCommonColor import xsigmaColorSeries
>>> xsigmaColorSeries.COOL
2
>>> isinstance(xsigmaColorSeries.ColorSchemes, xsigmaColorSeries.COOL)
>>> cs = xsigmaColorSeries()
>>> cs.SetColorScheme(xsigmaColorSeries.COOL)

Enum classes are wrapped in a manner similar to named enums, except that the enum values are placed within the enum class namespace. For example, xsigmaEventDataAction is an enum class, with ‘Press’ as a member:

>>> from xsigmamodules.xsigmaCommonCore import xsigmaEventDataAction
>>> xsigmaEventDataAction.Press
1
>>> isinstance(xsigmaEventDataAction.Press, xsigmaEventDataAction)
True

In the first example, the ColorSchemes enum type and the COOL enum value were both defined in the xsigmaColorSeries namespace. In the second example, the xsigmaEventDataAction enum class was defined in the module namespace, and the Press value was defined in the enum class namespace.

Note that the XSIGMA enum types behave like C++ enums, and not like the Python enums types provided by the Python ‘enum’ module. In particular, all XSIGMA enum values can be used anywhere that an int can be used.

Namespaces#

Namespaces are currently wrapped in a very limited manner. The only namespace members that are wrapped are enum constants and enum types. There is no wrapping of namespaced classes or functions, or of nested namespaces. Currently, the wrappers implement namespaces as Python module objects.

Docstrings#

The wrappers automatically generate docstrings from the doxygen comments in the header files. The Python help() command can be used to print the documentation to the screen, or the __doc__ attributes of the classes and methods can be accessed directly.

Internals and Advanced Topics#

Special Attributes#

Classes and objects derived from xsigmaObjectBase have special attributes, which are only used in very special circumstances.

The __xsigmaname__ attribute of the class provides the same string that the GetClassName() method returns. With the exception of classes that are template instantiations, it is identical to the __name__ attribute. For template instantiations, however, GetClassName() and __xsigmaname__ return the result of calling typeid(cls).name() from C++, which provides a platform specific result:

>>> xsigmaSOADataArrayTemplate['float32'].__xsigmaname__
'23xsigmaSOADataArrayTemplateIfE'

This can be used to get the XSIGMA ClassName when you don’t have an instantiation to call GetClassName() on. It is useful for checking the type of a C++ XSIGMA object against a Python XSIGMA class.

The __this__ attribute of the objects is a bit less esoteric, it provides a pointer to the C++ object as a mangled string:

>>> a = xsigmaFloatArray()
>>> a.__this__
'_00005653a6a6f700_p_xsigmaFloatArray'

The string provides the hexadecimal address of ‘this’, followed by ‘p’ (shorthand for pointer), and the type of the pointer. You can also construct a Python object directly from the C++ address, if the address is formatted as described above:

>>> a = xsigmaFloatArray('_00005653a6a6f700_p_xsigmaFloatArray')
>>> a
<xsigmamodules.xsigmaCommonCore.xsigmaFloatArray(0x5653a6a6f700) at 0x7f0e7aecf5e0>

If you call the constructor on the string provided by __this__, you will get exactly the same Python object back again, rather than a new object. But this constructor can be useful if you have some XSIGMA code that has been wrapped with a different wrapper tool, for example with SWIG. If you can get the XSIGMA pointer from SWIG, you can use it to construct Python object that can be used with the native XSIGMA wrappers.

Wrapper Hints#

A wrapper hint is an attribute that can be added to a class, method, or parameter declaration in a C++ header file to give extra information to the wrappers. These hints are defined in the xsigmaWrappingHints.h header file.

The following hints can appear before a method declaration:

  • XSIGMA_WRAPEXCLUDE excludes a method from the wrappers

  • XSIGMA_NEWINSTANCE passes ownership of a method’s return value to the caller

For convenience, XSIGMA_WRAPEXCLUDE can also be used to exclude a whole class. The XSIGMA_NEWINSTANCE hint is used when the return value is a xsigmaObjectBase* and the caller must not increment the reference count upon acceptance of the object (but must still decrement the reference count when finished with the object).

The following hints can appear after a method declaration:

  • XSIGMA_EXPECTS(cond) provides preconditions for the method call

  • XSIGMA_SIZEHINT(expr) marks the array size of a return value

  • XSIGMA_SIZEHINT(name, expr) marks the array size of a parameter

For XSIGMA_EXPECTS(cond), the precondition must be valid C++ code, and can use any of the parameter names or this. Even without this, any public names in the class namespace (including method names) will be resolved. See the Preconditions section for additional information.

XSIGMA_SIZEHINT(expr) is used for methods that return an array as type T*, where T is a numeric data type. The hint allows the wrappers to convert the array to a tuple of the correct size. Without the size hint, the wrappers will return the pointer as a string that provides a mangled memory address of the form ‘_hhhhhhhhhhhh_p_void’ where ‘hhhhhhhhhhhh’ is address expressed in hexadecimal.

XSIGMA_SIZEHINT(parameter_name, expr) is used to hint parameters of type T* or T&* (with T as a numeric data type) so that the wrappers know the size of the array that the pointer is pointing to. The expr can be any expression that evaluates to an integer, and it can include parameter names, public class members and method calls, or the special name _ (underscore) which indicates the method’s return value. In the absence of a size hint, the wrappers cannot check that the length of the sequence passed from Python matches the size of the array required by the method. If the method requires a larger array than it receives, a buffer overrun will occur.

The following hints can appear before a parameter declaration:

  • XSIGMA_FILEPATH marks a parameter that accepts a pathlib.Path object

  • XSIGMA_ZEROCOPY marks a parameter that accepts a buffer object

More specifically, XSIGMA_FILEPATH is used with char* and std::string parameters to indicate that the method also accepts any object with a __fspath__() method that returns a path string. And XSIGMA_ZEROCOPY is used with T* parameters, for basic integer or float type T, to indicate that the Python buffer protocol will be used to access the values, rather than the Python sequence protocol that is used by default.

Deprecation Warnings#

In addition to the wrapping hints, the Python wrappers are also aware of the deprecation attributes that have been applied to classes and methods. When a deprecated method is called, a DeprecationWarning is generated and information about the deprecation is printed, including the XSIGMA version for the deprecation.

To ignore these warnings, use the following code:

import warnings
warnings.filterwarnings('ignore', category=DeprecationWarning)

To see each deprecation warning just once per session,

warnings.filterwarnings('once', category=DeprecationWarning)

Exception Handling#

There are times when an observer might generate a Python exception. Since the observers are called from C++, there is no good way to catch these exceptions from within Python. So, instead, the wrappers simply print a traceback to stderr and then clear the error indicator. The Python program will continue running unless the exception was a KeyboardInterrupt (Ctrl-C), in which case the program will exit with an error code of 1.

Deleting a xsigmaObject#

There is no direct equivalent of XSIGMA’s Delete() method, since Python does garbage collection automatically. The Python object will be deleted when there are no references to it within Python, and the C++ object will be deleted when there are no references to it from within either Python or C++. Note that references can hide in unexpected places, for example if a method of an object is used as an observer callback, the object will not be deleted until the observer is disconnected.

The DeleteEvent can be used to detect object deletion, but note that the observer will receive a None for the object, since the observer is called after (not before) the deletion occurs:

>>> o = xsigmaObject()
>>> o.AddObserver('DeleteEvent', lambda o,e: print(e, o))
1
>>> del o
DeleteEvent None

If you need to know what object is deleted, the identifying information must be extracted before the deletion occurs:

>>> o = xsigmaObject()
>>> o.AddObserver('DeleteEvent',lambda x,e,r=repr(o): print(e, r))
1
>>> del o
DeleteEvent <xsigmamodules.xsigmaCommonCore.xsigmaObject(0x55783870f970) at 0x7f1e61678be0>

In cases where you need to track down tricky memory issues, you might find it useful to call the GetReferenceCount() method of the object directly.

Ghosts#

A wrapped XSIGMA object (derived from xsigmaObjectBase) is a Python object that holds a pointer to a C++ object (specifically, a xsigmaObjectBase*). The Python object can have attributes that the C++ object knows nothing about. So, what happens to these attributes if the Python object is deleted, but the C++ object lives on? Consider this simple example of storing the C++ object in an array and then deleting the Python object:

obj = xsigmaObject()
obj.tag = 'FirstObject'
va = xsigmaVariantArray()
va.InsertNextValue(obj)
del obj

When we retrieve the object from the array, we want it to have the ‘tag’ attributes that it had we stored it. But you might wonder, aren’t all Python-specific attributes deleted along with the Python object? The answer is, no they aren’t, they’re saved until until the C++ object itself is deleted.

The wrappers have a special place, which we will call the graveyard, where ‘ghosts’ of objects are stored when the objects are deleted. The ghost is not an object, but rather a container for the Python attributes of a deceased object. If the object ever reappears within Python, usually as a return value from a C++ method call, then the ghost is resurrected as a new Python object that has all the attributes of the original Python object.

The graveyard is only used for objects that have unfinished business. If a Python object has an empty dict and no other special attributes, then it will not go to the graveyard. Also, if the C++ object is deleted at the same time as the Python object, then the graveyard will not be used. Each ghost in the graveyard holds a weak pointer to its C++ object and will vanish when the C++ object is deleted (not immediately, but the next time the graveyard garbage collector runs).

Subclassing a XSIGMA Class#

It is possible to subclass a XSIGMA class from within Python, but this is of limited use because the C++ virtual methods are not hooked to the Python methods. In other words, if you make a subclass of xsigmaPolyDataAlgorithm and override override the Execute() method, it will not be automatically called by the XSIGMA pipeline. Your Execute() method will only be called if the call is made from Python.

The addition of virtual method hooks to the wrappers has been proposed, but currently the only way for Python methods to be called from C++ code is via callbacks. The xsigmaProgrammableSource and xsigmaProgrammableFilter are examples of XSIGMA algorithm classes that use callbacks for execution, while xsigmaInteractionStyleUser can use observer callbacks for event handling.

Wrapping External XSIGMA Modules#

If you have your own C++ classes that are based on XSIGMA, and if they are placed with a XSIGMA module with a xsigma.module file, then they can be wrapped as shown in the Module Wrapping Example. You will also find the cmake documentation on XSIGMA modules to be useful.

Experimental Features#

Python Class Overrides#

XSIGMA now supports overriding wrapped classes with Python subclasses. This enables developers to provide more Python friendly interfaces for certain classes. Here is a trivial example of an override:

from xsigmamodules.xsigmaCommonCore import xsigmaPoints
@xsigmaPoints.override
class CustomPoints(xsigmaPoints):
    pass

Once the override is in place, any future xsigmaPoints Python object instances will be instances of the override class. This behavior is global.

points = xsigma.xsigmaPoints() # returns an instance of CustomPoints

The override can be reversed by setting an override of None, but this will not impact instantiations that have already occurred.

xsigmaPoints.override(None)

If the class has already been overridden in C++ via XSIGMA’s object factory mechanism, then directly applying a Python override to that class will not work. Instead, the Python override must be applied to the C++ factory override. For example, on Windows,

@xsigmaWin32OpenGLRenderWindow.override
class CustomRenderWindow(xsigmaWin32OpenGLRenderWindow):
    ...
window = xsigmaRenderWindow() # creates a CustomRenderWindow

Please see Subclassing a XSIGMA Class for restrictions on subclassing XSIGMA classes through Python.

Stub Files for Type Hinting#

XSIGMA includes a script called generate_pyi.py that will generate pyi stub files for each wrapped XSIGMA module. The purpose of these files, as explained in PEP 484, is to provide type information for all constants, classes, and methods in the modules. Each of these files contain blocks like this:

XSIGMA_DOUBLE:int
XSIGMA_DOUBLE_MAX:float
XSIGMA_DOUBLE_MIN:float
...

class xsigmaObject(xsigmaObjectBase):
    def AddObserver(self, event:int, command:Callback, priority:float=0.0) -> int: ...
    def GetMTime(self) -> int: ...
    @staticmethod
    def GetNumberOfGenerationsFromBaseType(type:str) -> int: ...
    @overload
    def HasObserver(self, event:int, __b:'xsigmaCommand') -> int: ...
    @overload
    def HasObserver(self, event:str, __b:'xsigmaCommand') -> int: ...

class xsigmaAbstractArray(xsigmaObject):
    class DeleteMethod(int): ...
    XSIGMA_DATA_ARRAY_ALIGNED_FREE:'DeleteMethod'
    XSIGMA_DATA_ARRAY_DELETE:'DeleteMethod'
    XSIGMA_DATA_ARRAY_FREE:'DeleteMethod'
    XSIGMA_DATA_ARRAY_USER_DEFINED:'DeleteMethod'
    def Allocate(self, numValues:int, ext:int=1000) -> int: ...

Python consoles like ipython and IDEs like PyCharm can use the information in these files to provide hints while you edit the code. These files are included in the Python packages for XSIGMA, but they can also be built by executing the generate_pyi.py script. To do so, execute the script with the xsigmapython executable (or with the regular python executable, if its paths are set for XSIGMA):

xsigmapython -m xsigmamodules.generate_pyi

This will place build the pyi files and place them inside the xsigmamodules package, where ipython and PyCharm should automatically find them. The help for this script is as follows:

usage: python generate_pyi.py [-p package] [-o output_dir] [module ...]
options:
  -p NAME        Package name [xsigmamodules by default].
  -o OUTPUT      Output directory [package directory by default].
  -e EXT         Output file suffix [.pyi by default].
  module         Module or modules to process [all by default].

The pyi files are syntactically correct python files, so it is possible to load them as such in order to test them and inspect them.
